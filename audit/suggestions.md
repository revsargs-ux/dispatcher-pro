# Dispatcher.PRO — Feature Suggestions & Improvements
**Дата:** 2026-05-14  
**Источник:** Полный аудит (UI, Security, Code Quality, Supabase, Notifications, Integration)

---

## 1. Security Improvements

### 1.1 Row Level Security на всех таблицах
**Описание:** Включить RLS на всех 16 таблицах Supabase. Создать политики: service_role — полный доступ, anon — только через аутентификацию. Пароли никогда не возвращать через REST API.
**Почему важно:** Сейчас любой с anon key может читать пароли, изменять данные, удалять записи. Критический риск.
**Сложность:** medium

### 1.2 Content-Security-Policy Header
**Описание:** Добавить CSP header в cors.js: `default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; connect-src 'self' https://*.supabase.co`. Настроить постепенно (report-only → enforce).
**Почему важно:** Без CSP XSS атака может загрузить внешние скрипты, украсть токены, перенаправить пользователей.
**Сложность:** small

### 1.3 HMAC подпись для GAS Webhook
**Описание:** Заменить статический secret `dp_gas_sync_2026` на HMAC-SHA256 подпись тела запроса. GAS генерирует подпись с shared secret, сервер верифицирует.
**Почему важно:** Статический secret в заголовке можно перехватить и использовать для записи любых данных в БД.
**Сложность:** small

### 1.4 Верификация при привязке бота
**Описание:** При `/start <phone>` в TG/МАКС — отправить SMS/код подтверждения на указанный номер. Привязка только после ввода кода.
**Почему важно:** Сейчас кто угодно может привязать чужой номер к своему Telegram, получив доступ к данным.
**Сложность:** medium

### 1.5 Убрать plaintext fallback в checkPassword
**Описание:** Удалить строку `if (storedPassword === inputPassword) return true` из auth.js. Мигрировать все plaintext пароли в bcrypt. Принудительный сброс пароля для затронутых пользователей.
**Почему важно:** Plaintext сравнение означает что любой с доступом к БД может залогиниться как любой пользователь.
**Сложность:** small

### 1.6 Refresh Token Rotation
**Описание:** При каждом refresh — инвалидировать старый refresh token и выдавать новый. Хранить "token family" на сервере для обнаружения кражи.
**Почему важно:** Сейчас украденный JWT валиден 7 дней. Refresh rotation позволяет обнаружить что старый токен использован повторно.
**Сложность:** medium

## 2. UX Improvements

### 2.1 Уведомление о завершении смены
**Описание:** Добавить событие "смена завершена" — уведомлять рабочего (если он завершил) и клиента через все каналы (TG + МАКС + Push). Отправлять сводку: дата, часы, сумма к оплате.
**Почему важно:** Сейчас ни один канал не уведомляет о завершении. Клиент не знает что работа закончилась.
**Сложность:** medium

### 2.2 Кнопка "Завершить все смены" для диспетчера
**Описание:** Mass action для завершения всех активных смен дня с расчётом часов и отправкой уведомлений.
**Почему важно:** При 10+ рабочих — вручную закрывать каждую смену долго. Один клик вместо 10.
**Сложность:** small

### 2.3 Phone input с маской + валидация
**Описание:** Добавить маску ввода телефона (+7 ___ ___-__-__) во все формы (login, register, add worker/client). Валидация 11 цифр на frontend И backend.
**Почему важно:** worker.html валидирует, остальные — нет. Пользователи вводят телефоны в разных форматах.
**Сложность:** small

### 2.4 Escape key закрывает модалки
**Описание:** Добавить `document.addEventListener('keydown', e => { if (e.key === 'Escape') closeModal() })` во все файлы. Focus trap в открытых модалках.
**Почему важно:** Стандартное UX ожидание. Сейчас Escape не работает, нужно тянуться к ×.
**Сложность:** small

### 2.5 Push-уведомления — починить или убрать UI
**Описание:** Либо развернуть Edge Function с корректной encryption (web-push библиотека), либо убрать подписку на push из UI чтобы не вводить в заблуждение.
**Почему важно:** Push кнопка есть, но уведомления никогда не доходят. Пользователь думает что всё настроено.
**Сложность:** medium (deploy Edge Function) / small (убрать)

### 2.6 Offline-режим с Service Worker
**Описание:** Расширить sw.js для кеширования статических ресурсов и базового UI. Показывать "Нет подключения" вместо белого экрана.
**Почему важно:** Рабочие на объектах часто без интернета. App выглядит сломанным.
**Сложность:** medium

### 2.7 Тёмная тема
**Описание:** Добавить `@media (prefers-color-scheme: dark)` CSS с тёмными цветами. Переключатель в настройках.
**Почему важно:** Ночные смены — яркий белый экран режет глаза. Частый запрос мобильных пользователей.
**Сложность:** small

## 3. Performance

### 3.1 Batch запросы вместо N+1
**Описание:** Заменить циклы с fetch на batch запросы: `worker_id=in.(id1,id2,...)`. handlePostProcess — использовать Supabase joins `select=*,clients(name),service_types(name)`. handleTrackingWorkersLocation — один запрос вместо N.
**Почему важно:** Один POST /shifts = 8-12 API вызовов. При 100+ смен в день — 800-1200 запросов. Supabase latency × 12 = медленный ответ.
**Сложность:** medium

### 3.2 Page Visibility API для polling
**Описание:** Обернуть `setInterval(loadDashboard, 5000)` в проверку `document.hidden`. Ставить на паузу когда вкладка неактивна, возобновлять при `visibilitychange`.
**Почему важно:** 5-7 API вызовов каждые 5 секунд в фоне = ~60 запросов/мин на пустом месте. Нагрузка на сервер и трафик.
**Сложность:** small

### 3.3 Debounce для поиска
**Описание:** Добавить debounce (300ms) на worker-search и client-search inputs в index.html и owner.html. Сейчас каждый символ = API запрос.
**Почему важно:** Быстрый набор = 10-15 запросов за секунду. Supabase rate limiting может сработать.
**Сложность:** small

### 3.4 Кешировать service_types на клиенте
**Описание:** service_types меняются редко. Загрузить один раз при логине, кешировать в sessionStorage. Обновлять раз в час.
**Почему важно:** service_types запрашивается при каждом открытии модалки смены. 10 смен = 10 запросов к одним и тем же данным.
**Сложность:** small

### 3.5 Оптимизировать loadOverview / loadDashboard
**Описание:** Объединить 5-7 параллельных запросов в 1-2 batch запроса с aggregates. Shifts count, payments sum, active workers — всё можно вычислить одним Supabase запросом с `select=count` и groupBy.
**Почему важно:** Каждый poll = 5-7 запросов. При 10 открытых вкладках = 50-70 запросов каждые 5 секунд.
**Сложность:** medium

## 4. Architecture

### 4.1 Разбить routes.js на модули
**Описание:** routes.js (863 строк) → `routes/auth.js`, `routes/tracking.js`, `routes/api-proxy.js`, `routes/admin.js`, `routes/index.js` (router).
**Почему важно:** Монолит трудно поддерживать и тестировать. Изменение в tracking может сломать auth.
**Сложность:** medium

### 4.2 Общий модуль для ботов (bot-common.js)
**Описание:** Вынести общую логику telegram.js + max-bot.js (~400 строк): команды, AI запрос, расчёт зарплат, ключевые слова. Адаптеры отправки — тонкие обёртки.
**Почему важно:** 80% дублирования. Исправление бага в одном боте — надо копировать в другой. Забыл скопировать = баг.
**Сложность:** medium

### 4.3 Общий db helper (sbFetch)
**Описание:** sbHeaders/sbFetch дублирован в 5 местах. Вынести в `modules/db.js` с единым API: `db.fetch(table, query)`, `db.insert(table, data)`, `db.update(table, id, data)`.
**Почему важно:** Изменение формата заголовков или URL — надо править 5 файлов.
**Сложность:** small

### 4.4 Удалить весь мусор (.bak, backups/)
**Описание:** `rm -rf *.bak modules/*.bak backups/`. 80+ файлов, ~2MB мёртвого кода.
**Почему важно:** Затрудняет поиск по коду, путает при редактировании. Git = бэкапы.
**Сложность:** small

### 4.5 Migration tracking
**Описание:** Создать таблицу `schema_migrations (version text PK, applied_at timestamptz)`. Простой скрипт для применения миграций по порядку.
**Почему важно:** Сейчас миграции применяются вручную. Риск пропустить или применить дважды при деплое на новое окружение.
**Сложность:** small

### 4.6 Заменить notifications.json на Supabase таблицу
**Описание:** Перенести in-app уведомления из JSON файла в Supabase таблицу с TTL (автоудаление через 30 дней). Привязать к конкретному пользователю.
**Почему важно:** JSON файл растёт бесконечно, конкурентная запись без блокировок, все видят одни уведомления.
**Сложность:** small

## 5. New Features

### 5.1 Мультиязычность (RU/EN)
**Описание:** Вынести все строки в языковой файл. Определять язык по браузеру / настройке пользователя. Начать с worker.html и client.html (самые клиентские).
**Почему важно:** Потенциальные клиенты могут быть экспатами. Интернационализация расширяет аудиторию.
**Сложность:** large

### 5.2 Фото-отчёт с объекта
**Описание:** Worker может сделать фото до/после работы → загрузить через бота или web → прикрепляется к смене → видно клиенту.
**Почему важно:** Подтверждение выполненной работы. Снижает споры между клиентом и рабочим.
**Сложность:** medium

### 5.3 Автоматический расчёт зарплаты
**Описание:** По завершении смены — auto-расчёт: часы × ставка + доп.услуги. Отправка сводки рабочему и клиенту. Подтверждение обеими сторонами.
**Почему важно:** Сейчас расчёт ручной. Ошибки в подсчёте = конфликты. Автоматизация экономит время диспетчера.
**Сложность:** medium

### 5.4 Dashboard с графиками (Dispatcher/Owner)
**Описание:** Интерактивные графики: доходы по дням/неделям, загрузка рабочих, топ клиентов. Использовать Chart.js или встроенный Canvas (уже есть в owner.html).
**Почему важно:** owner.html уже имеет canvas chart, но данные ограничены. Полный dashboard = основа для принятия решений.
**Сложность:** medium

### 5.5 Рейтинги и отзывы
**Описание:** После завершения смены — клиент ставит оценку рабочему (1-5 звёзд + опциональный комментарий). Worker видит свой рейтинг. Dispatcher видит рейтинги при назначении.
**Почему важно:** Мотивация рабочих к качеству. Dispatcher может выбирать лучших. Клиент видит надёжность.
**Сложность:** medium

### 5.6 Интеграция с Google Calendar / iCal
**Описание:** Экспорт смен в Google Calendar. Подписка на iCal feed — автоматическое обновление при изменении смен.
**Почему важно:** Клиенты ведут свои календари. Дублирование = ошибки. Интеграция = синхронизация.
**Сложность:** medium

### 5.7 Chat между клиентом и рабочим (через бота)
**Описание:** Клиент может написать сообщение рабочему через TG/МАКС бота (анонимно, через бота). Рабочий отвечает тоже через бота. Dispatcher видит всю переписку.
**Почему важно:** Координация на объекте (опоздание, уточнение адреса). Сейчас — звонки через диспетчера, лишнее звено.
**Сложность:** large

### 5.8 Повторяющиеся заказы
**Описание:** Клиент может создать шаблон заказа (каждый понедельник, одинаковые услуги). Автоматическое создание смен по расписанию.
**Почему важно:** Постоянные клиенты = 80% заказов. Сейчас диспетчер создаёт каждую смену вручную.
**Сложность:** medium

### 5.9 Telegram Web App вместо worker.html / client.html
**Описание:** Заменить отдельные HTML страницы на Telegram Web App (Mini App). Пользователь открывает из бота — нативный опыт в Telegram.
**Почему важно:** Рабочие уже в TG. Лишний шаг — открывать браузер. Mini App = мгновенный доступ, авторизация через TG.
**Сложность:** large

### 5.10 Двухфакторная аутентификация
**Описание:** При логине — SMS код или TG/МАКС бот подтверждение. Особенно для owner и dispatcher ролей.
**Почему важно:** Финансовые данные, управление рабочими. Один пароль = единственная защита.
**Сложность:** medium
